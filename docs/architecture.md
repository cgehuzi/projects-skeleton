# Архитектура

Документ фиксирует принципиальные решения. Он первичен по отношению к коду:
расхождение кода с этим документом — повод исправить код или обновить документ.

## 1. Слои

- **backend** — Laravel 13. Совмещает три роли:
  1. **API** для фронтенда (REST/JSON под префиксом `/api`).
  2. **CMS** — админка на Filament 4 (`/admin`), где редакторы управляют
     контентом и маршрутами.
  3. **Источник истины по маршрутам** — таблица `pages` (slug → тип → блоки).
- **frontend** — Next.js 16. Headless SSR-слой презентации.

> Каркасы `backend/`/`frontend/` в skeleton пустые и создаются `make init` (Model B).
> Начинка (модели/контроллеры/блоки/дизайн-система, в т.ч. `pages` и render-эндпоинт)
> подключается зависимостью `package-core`, а не входит в skeleton.

## 2. Потоки

Важно различать два направления:

- **Публикация контента** (авторский): `Filament → БД → API → SSR`.
- **Обработка запроса** (runtime): `Браузер → Next (SSR) → API → БД`.

## 3. Backend-driven routing

Публичный роутинг — не в Next, а в бэкенде.

- В Next один catch-all сегмент `app/[locale]/[[...slug]]/page.tsx`.
- Он вызывает render-эндпоинт API: `GET /api/render?path=/...&locale=ru`.
- Эндпоинт за **один** запрос возвращает контракт страницы:

  ```jsonc
  {
    "status": 200,            // 200 | 301 | 302 | 404
    "redirect": null,         // целевой URL при 301/302
    "locale": "ru",
    "route": { "type": "page", "id": 42 },
    "seo": { "title": "...", "description": "...", "canonical": "..." },
    "blocks": [               // блочная модель (формат Filament Builder)
      { "type": "hero", "data": { /* ... */ } },
      { "type": "text", "data": { /* ... */ } }
    ]
  }
  ```

- Роутинг Next **не выключается** (это невозможно — SSR идёт через роутер), а
  **сжимается** до catch-all. В Next остаются только служебные маршруты:
  `revalidate`, `preview`, `sitemap.ts`, `robots.ts`, `health`, `middleware`.

## 4. Блочная модель и component resolver

- Бэкенд отдаёт нормализованный массив `blocks` (`type` + `data`, формат Filament Builder).
- На фронте — реестр `type → React-компонент`. Неизвестный тип не роняет
  страницу (рендерится заглушка/пропускается).
- Так редактор собирает страницы из блоков, а фронт остаётся развязанным от
  конкретной контент-модели.

## 5. Кэширование и инвалидация

Самая ответственная часть архитектуры — проектируется с первого дня.

- В Next — ISR с **cache tags** (тег на сущность/маршрут).
- В Filament — обсерверы моделей при сохранении дёргают
  `POST /api/revalidate` (или Next route handler `/api/revalidate`) с тегами
  изменённого контента → точечный сброс кэша, без устаревших страниц.

## 6. Мультиязычность

- Язык по умолчанию — `ru`.
- Локаль — префикс пути (`/ru/...`, `/en/...`); дефолтная локаль может идти без
  префикса (решается в middleware/резолвере).
- Резолвер маршрутов учитывает локаль; slug уникален в рамках локали.

## 7. Сетевое взаимодействие

- Внешняя маршрутизация — общий **Traefik** (один на машину) по хосту:
  `<project>.localhost` → Next, `api.<project>.localhost` → Laravel. Проекты
  работают параллельно без конфликтов портов. См. `docker/proxy/`.
- Server-side обращения Next к API идут **внутри сети проекта** по
  `http://api-internal/api` (`INTERNAL_API_URL`) — без публичного хопа.
  `api-internal` — алиас nginx в project-scoped сети `app` (имя `nginx` нельзя:
  конфликтует в общей сети `proxy`).
- Браузерные обращения (если понадобятся) — на `http://api.<project>.localhost/api`
  (`NEXT_PUBLIC_API_URL`).
- Порты Postgres/Redis на хост — эфемерные (loopback), смотреть `make ports`.

## 8. Открытые вопросы (решаются по мере роста)

- Preview/draft mode: токены из Filament + Next Draft Mode.
- Типобезопасность контракта: OpenAPI из Laravel → генерация TS-типов.
- Управление редиректами (301/302) через Filament.
- `sitemap.xml` / `robots.txt` из данных бэкенда.
